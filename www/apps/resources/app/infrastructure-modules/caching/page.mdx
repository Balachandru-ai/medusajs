# {metadata.title}

In this guide, you'll learn about the Caching Module and its providers.

<Note>

The Caching Module is available starting [Medusa v2.11.0](https://github.com/medusajs/medusa/releases/tag/v2.11.0). It replaces the deprecated [Cache Module](../cache/page.mdx).

</Note>

## What is the Caching Module?

The Caching Module provides functionality to cache data in your Medusa application, improving performance and reducing latency for frequently accessed data.

For example, Medusa uses the Caching Module to cache product information. You can also cache custom data such as brand information.

The Caching Module stores and retrieves cached data using the caching service you've integrated, such as Redis or Memcached. This provides flexibility in customizing your Medusa application's infrastructure to meet your performance and scalability requirements.

### Caching Module vs Cache Module

Before Medusa v2.11.0, you used the [Cache Module](../cache/page.mdx) to cache data. The Cache Module is now deprecated and has been replaced by the Caching Module.

Your Medusa application now uses the Caching Module by default with the [Redis Caching Module Provider](./providers/redis/page.mdx). If you were using the Cache Module in your application, you should remove it from your `medusa-config.ts` and migrate to the Caching Module instead.

<Note>

You must configure the environment variable for Redis, as explained in the [Redis Caching Module Provider](./providers/redis/page.mdx#default-redis-configuration) guide. Otherwise, your Medusa application will fall back to the [In-Memory Caching Module Provider](./providers/in-memory/page.mdx), which is not suitable for production.

</Note>

## How to Use the Caching Module

There are two ways to cache data with the Caching Module: when retrieving data with the [Query](!docs!/learn/fundamentals/module-links/query) and [Index Module](!docs!/learn/fundamentals/module-links/index-module), or by directly using the Caching Module's service.

### 1. Caching with Query or Index Module

You can cache results of Query and the Index Module by passing the `cache` option to the `query.graph` and `query.index` methods, or to the `useQueryGraphStep` in workflows.

For example:

```ts highlights={[["14"], ["15"], ["16"]]}
import {
  createWorkflow,
  WorkflowResponse
} from "@medusajs/framework/workflows-sdk"
import { useQueryGraphStep } from "@medusajs/medusa/core-flows"

export const workflow = createWorkflow(
  "workflow-1",
  () => {
    const { data: products } = useQueryGraphStep({
      entity: "product",
      fields: ["id", "title"],
      options: {
        cache: {
          enable: true,
        },
      },
    })

    return new WorkflowResponse(products)
  }
)
```

The `useQueryGraphStep` accepts an `options.cache` property that allows you to enable and configure caching of the results.

When caching is enabled, the Caching Module stores the results in the underlying caching service (for example, Redis). Subsequent calls to the same query will retrieve the results from the cache, improving performance.

### 2. Caching with the Caching Module's Service

You can also directly use the Caching Module's service to cache custom data in your Medusa application.

For example, you can resolve the Caching Module's service in a workflow step and use its methods to set, retrieve, or delete cached data.

```ts highlights={[["16"], ["17"], ["18"]]}
import { Modules } from "@medusajs/framework/utils"
import { 
  createStep,
  createWorkflow,
  StepResponse,
  WorkflowResponse,
} from "@medusajs/framework/workflows-sdk"

const getBrandStep = createStep(
  "get-brand-step",
  async ({}, { container }) => {
    const cachingModuleService = container.resolve(
      Modules.CACHING
    )

    const cachedValue = await cachingModuleService.get({
      tags: ["brand"]
    })

    return new StepResponse(cachedValue)
  }
)

const workflow = createWorkflow(
  "workflow-1",
  () => {
    const brand = getBrandStep()

    // Do something with the brand
  }
)
```

In the example above, you create a step that resolves the Caching Module's service from the [Medusa container](!docs!/learn/fundamentals/medusa-container).

Then, you use the `get` method of the service to retrieve a cached value with the tag `"brand"`. You can also use other methods such as `set` to cache a value and `clear` to remove a cached value.

Caching data with Query or the Index Module is ideal for caching database query results. The Caching Module will:

1. Generate unique cache keys based on query parameters, ensuring different queries are cached separately.
2. Automatically invalidate cached data based on the specified time-to-live (TTL) or the global configuration of the [default Caching Module Provider](./providers/page.mdx#default-caching-module-provider).

The `cache` option is highly configurable, allowing you to customize the caching behavior for each query. For example, you can specify custom keys or disable automatic invalidation. Refer to the [Query guide](!docs!/learn/fundamentals/module-links/query#caching) for more details on the available options.

If you need to cache custom data that isn't directly related to database queries, such as computed values or external API responses, use the Caching Module's service directly. In this case, you'll need to manage cache keys yourself, though you can still enable automatic invalidation using the service's `set` method.

## What is a Caching Module Provider?

A Caching Module Provider implements the underlying logic for caching data, such as integrating third-party caching services. The Caching Module then uses the registered Caching Module Provider to handle caching operations.

Refer to the [Caching Module Providers guide](./providers/page.mdx) to learn more about Caching Module Providers in Medusa, and how to configure the default provider.
