# {metadata.title}

The In-Memory Caching Module Provider is a simple caching solution that stores cached data in the server's memory.

While it's suitable for development and testing environments, it's not recommended for production use due to its limitations in scalability and data persistence. Use the [Redis Caching Module Provider](../redis/page.mdx) instead.

<Note>

The Caching Module and its providers are available starting [Medusa v2.11.0](https://github.com/medusajs/medusa/releases/tag/v2.11.0).

</Note>

---

## Register the In-Memory Caching Module

To enable and configure the In-Memory Caching Module Provider, add the `in_memory` option to the Caching Module configuration in your `medusa-config.ts` file:

```ts title="medusa-config.ts"
module.exports = defineConfig({
  // ...
  modules: [
    {
      resolve: "@medusajs/medusa/caching",
      options: {
        in_memory: {
          enable: true,
          // other options...
        },
      }
    }
  ]
})
```

This will enable the In-Memory Caching Module Provider in your Medusa application. It will be used as the default provider if no other provider is set as default, as explained in the [Caching Module Providers guide](../page.mdx#default-caching-module-provider).

## Test the In-Memory Caching Module

You can test the In-Memory Caching Module by caching data using the Query or Index Module, or by directly using the Caching Module's service, as described in the [Caching Module guide](../page.mdx#how-to-use-the-caching-module).

If the In-Memory Caching Module Provider isn't the default provider, you can explicitly specify the In-Memory Caching Module Provider by using its provider ID, which is `cache-memory`, when caching data with Query, the Index Module, or directly with the Caching Module's service.

For example, you can create a workflow in `src/workflows/cache-products.ts` that caches products using the In-Memory Caching Module Provider:

```ts title="src/workflows/cache-products.ts" highlights={[["14"], ["15"], ["16"], ["17"]]}
import {
  createWorkflow,
  WorkflowResponse
} from "@medusajs/framework/workflows-sdk"
import { useQueryGraphStep } from "@medusajs/medusa/core-flows"

export const cacheProductsWorkflow = createWorkflow(
  "cache-products",
  () => {
    const { data: products } = useQueryGraphStep({
      entity: "product",
      fields: ["id", "title"],
      options: {
        cache: {
          enable: true,
          providers: ["cache-memory"],
        },
      },
    })

    return new WorkflowResponse(products)
  }
)
```

Next, execute that workflow in an [API route](!docs!/learn/fundamentals/api-routes). For example, create a route at `src/api/cache-product/route.ts` with the following content:

```ts title="src/api/cache-product/route.ts"
import { MedusaRequest, MedusaResponse } from "@medusajs/framework/http"
import { cacheProductsWorkflow } from "../../workflows/cache-products"

export const GET = async (req: MedusaRequest, res: MedusaResponse) => {
  const { result } = await cacheProductsWorkflow(req.scope)
    .run({})

  res.status(200).json(result)
}
```

Finally, start the Medusa application with the following command:

```bash npm2yarn
npm run dev
```

Then, make a `GET` request to the `/cache-product` endpoint:

```bash
curl http://localhost:9000/cache-product
```

You will receive a response with the list of products. The first time you make this request, the products will be fetched from the database and cached in memory. Subsequent requests will retrieve the products from the cache, which improves performance.
